## What this is

A static fantasy-football scoring site for the 2026 World Cup group stage.
Each participant fills in their predicted scores in a copy of the master Excel file.
The site reads those Excel files in the browser using SheetJS and scores them
against the real results in `data/master.xlsx`.

Owner: David Caicedo (`caicedodavid` on GitHub).
Repo: `git@github.com:caicedodavid/quiniela.git` (SSH over port 443 — see deployment).
Deployed on: Render (static site, auto-deploys on push to `main`).

## Stack

| Layer | Tool | Why |
|---|---|---|
| Hosting | Render static site | Free, auto-deploy on push |
| Styling | Tailwind CDN (Play) | No build step |
| Excel parsing | SheetJS 0.18.5 (vendored) | `vendor/xlsx.full.min.js` — CDN blocked on Walmart network |
| Build step | Python 3 + openpyxl | Pre-renders scores at deploy time |
| Local server | `bash serve.sh` | Auto-restarts on crash, auto-updates scores |

**No Node.js, no bundler, no framework.** Pure ES modules loaded directly by the browser.

## The Excel layout (CRITICAL — read this before touching parser.js or prerender.py)

The source file is `Excel-Mundial-2026.xlsx`, a heavily formula-driven workbook
with 9 sheets, all SHA-512 password protected. We stripped protection at the ZIP/XML
level and kept all sheets (helper sheets are required for formulas to resolve).

### Sheet: WORLDCUP

This is the only sheet the code reads. All other sheets are hidden.

**Group layout** — 12 groups (A–L), indexed `i = 0..11`:

```
Match rows  : row (4 + 8*i)  ..  (9 + 8*i)   — 6 rows, one per fixture
Team names  : row (5 + 8*i)  ..  (8 + 8*i)   — col A (1-based), 4 rows
Standings   : row (6 + 4*i)  ..  (9 + 4*i)   — col BD (56), 4 rows
```

**Columns used** (1-based for openpyxl, 0-based for SheetJS via `XLSX.utils.encode_cell`):

| Column | 1-based | 0-based | Content |
|--------|---------|---------|---------|
| A | 1 | 0 | Team name (in group block) |
| AA | 27 | 26 | Home team name (formula, cached) |
| AC | 29 | 28 | **Home goals** ← predictions go here |
| AD | 30 | 29 | **Away goals** ← predictions go here |
| AF | 32 | 31 | Away team name (formula, cached) |
| BD | 56 | 55 | Standings team name (formula, cached) |
| BM | 65 | 64 | Rank string e.g. "1A" (not used in scoring) |

**Important:** AC/AD contain raw numbers (predictions). AA/AF/BD are formula cells
whose cached values openpyxl reads via `data_only=True`. SheetJS reads `.v` (cached value).
Do NOT trust `.f` (formula text) for team names.

### Player name

`Home!C10` — formula that resolves to the player's name.
Falls back to filename (minus `.xlsx`) if cell is empty or contains the
placeholder string `"Nombre"`.

## Scoring rules

| Situation | Points |
|-----------|--------|
| Exact score (both goals correct) | 6 |
| Correct winner/draw + one goal count correct | 4 |
| Correct winner/draw, neither goal count correct | 3 |
| Wrong outcome but one goal count correct | 1 |
| Nothing correct | 0 |
| Match not yet played | null (shown as —) |
| No prediction made | 0 |

**Standings bonus** (applied when all 6 group matches are played):
`BONUS_PER_POSITION = 2` pts per correctly predicted final group position.
Constant lives in `js/scorer.js` AND is replicated in `prerender.py` — keep them in sync.

**Standings tiebreaker order:** points → goal difference → goals for → name (alphabetical).
This is computed fresh from AC/AD values, NOT read from the BM/BD cached standings.

## Architecture decisions & gotchas

### Why `window._parserModule`?
`scorer.js` needs `computeStandings` from `parser.js`, but `ui.js` imports `scorer.js`,
creating a circular ES module dependency. Solved by having `app.js` (the entry point)
expose `computeStandings` on `window._parserModule` after importing it. Hacky but works.

### Why SheetJS is vendored
`cdn.sheetjs.com` is blocked by Code Puppy's tool guard on the Walmart network.
The browser can actually reach it fine, but to avoid surprises: `vendor/xlsx.full.min.js`
is SheetJS 0.18.5 extracted from the npm tarball. The CDN serves 0.20.3 but the API
we use (`XLSX.read`, `XLSX.utils.encode_cell`) is identical across versions.

### Why no player cache in app.js
Early versions cached parsed player data in `playerCache`. Removed because:
1. Players edit their own files mid-game
2. Excel's atomic save (temp-file rename) makes fetches fail transiently
Instead: every click re-fetches both player + master with `fetchWithRetry` (3 attempts, 800ms back-off).

### Why fetch has `?t=Date.now()`
Python's `http.server` and browser both cache aggressively. Cache-busting timestamp
ensures stale Excel files are never served after an edit.

### Spaces in filenames
`"la lacra.xlsx"` — `encodeURIComponent()` in app.js handles the URL encoding.
Python's SimpleHTTPRequestHandler correctly decodes `%20` back to space when serving.
Do NOT rename files to remove spaces; just use `encodeURIComponent` consistently.

### The server dies when Excel saves
Excel on macOS does an atomic rename on save: writes temp file, deletes original,
renames temp. For ~500ms the file doesn't exist. `fetchWithRetry` covers this window.
`serve.sh` has an auto-restart loop so a server crash doesn't kill the page.

### Mobile layout
- `< md`: sticky green top bar with `<select id="mobile-player-select">` dropdown
- `≥ md`: fixed left sidebar with player list buttons
- Both are populated by `renderSidebar()` in `ui.js`
- Match tables use `overflow-x-auto` + `min-w-[520px]` for horizontal scroll on mobile

## Local development

```bash
cd /Users/d0c0b74/worldcup-fantasy
bash serve.sh          # scores players, starts server, auto-restarts on crash
# → http://localhost:7026
```

`serve.sh` runs `prerender.py` first (requires `openpyxl` — `pip install openpyxl`).

### Adding a participant
1. Drop their `name.xlsx` into `data/players/`
2. Restart `serve.sh` (it auto-regenerates `scores.json`)
3. `git add -A && git commit -m "add: name" && git push`

### Updating real results
Edit `data/master.xlsx` → fill AC/AD for played matches → save → restart `serve.sh`.
